## 1. 研究のコアクレーム（1段落）

LLMコーディングエージェントは「掟上今日子問題」——コンテキストウィンドウが切れるたびに全てを忘れる——を抱えている。既存のメモリ系研究（MemGPT、Mem0、A-MAC等）は汎用会話エージェントを対象としており、コーディングエージェント特有のPTYレベルの行動シグナル（インタラプト・rewind）を活用していない。本研究はこのギャップを埋める。**Associative Context Memory (ACM)** は、インタラプト＋直後の会話を最強の失敗シグナルとして、成功・失敗体験を外部メモリに永続化するMCPサーバーとして実装し、コンテキストウィンドウサイズへの依存を低減する。

---

## 2. 差別化ポイント（競合との比較）

| システム | クロスセッション記憶 | 失敗体験 | PTY行動シグナル | コーディング特化 |
|---------|-----------------|---------|--------------|--------------|
| MemGPT / Letta | ✓ | ✗ | ✗ | ✗ |
| Mem0 / A-MEM | ✓ | ✗ | ✗ | ✗ |
| A-MAC (2026/03/04) | ✓ | 部分的 | ✗ | ✗ |
| REMEMBERER | ✓ | ✓ (Q値) | ✗ | ✗ |
| Self-Gen ICE | ✓ | ✗ | ✗ | ✗ |
| Letta Code | ✓ | ✗ | ✗ | ✓ |
| **ACM（本研究）** | **✓** | **✓** | **✓** | **✓** |

**最重要の差別化：** PTYインタラプト＋直後の会話を「最強の失敗シグナル」として定義した研究はまだない。

## 3. フィードバックシグナル強度序列（本研究の核心）

```
強い ─── インタラプト（Ctrl+C / Esc連打）+ 直後N=3〜5ターンの会話
             → 「やめてほしかった」の最明確シグナル
             → 直後の会話に「なぜ止めたか」が出る
         ↓
         rewind（/rewind または Esc×2）
         ↓
         修正指示テキスト（「違う」「やり直して」等）
         ↓
         手戻りなし完了 + テスト通過
──── 弱い
```

## 4. リサーチクエスチョン（5つ）

- **RQ1**: ACMはContext Rotを抑制し、タスク性能を向上させるか？
- **RQ2**: 成功のみ / 失敗のみ / 両方 — どの組み合わせが最も効果的か？
- **RQ3**: シグナル強度（インタラプト/rewind/修正指示）はメモリ品質と相関するか？
- **RQ4**: ACMはコンテキストウィンドウサイズへの依存を低減するか？（制限条件での測定）
- **RQ5**: SerenaとACMは相補的か？

## 9. 未解決の問い（後回しにしていいもの）

- インタラプト後キャプチャのターン数N の最適値
- 連想メモリの「忘却」設計（古い体験が逆効果になるケース）
- プロジェクト固有 vs グローバルメモリの分離
- 失敗体験の蓄積しすぎ問題（過学習的な挙動）
- プライバシー（体験エントリにコードが含まれる場合）

## 10. 意思決定ログ（なぜこの設計になったか）

| 決定 | 理由 |
|-----|-----|
| MCPサーバーとして実装 | Claude Code / Codex エコシステムに乗る。認められやすい。 |
| モデル非依存設計 | 特定APIに依存しない。Claude制約があればLocalLLMにフォールバック。 |
| 失敗体験を含める | 「やるな」の記憶が長期的な活動品質に直結する。成功のみでは不十分。 |
| インタラプトを最強シグナルに | PTYレベルで取れる最明確な否定シグナル。直後の会話がラベルになる。 |
| arXivを先に出す | A-MACが3日前に出た。時間との戦い。旗を立てることを優先。 |
| 学会後にAnthropicへ提案 | 査読を通した信頼性を持った上でアプローチする方が説得力がある。 |
