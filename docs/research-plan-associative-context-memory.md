# Associative Context Memory: LLMエージェントにおける連想記憶モデルの提案と評価

**作成日:** 2026-03-07  
**著者:** Yamato Hiroshi / Signal compose Inc.

---

## 1. 研究背景

LLMのコンテキストウィンドウは有限であり、トークン量が増加するにつれてモデルの性能が劣化することが実証されている（Context Rot, Chroma Research 2025）。この問題に対する既存のアプローチは主に以下の2系統に分類される：

**圧縮・分割系（対症療法）**
- auto-compact（会話要約による引き継ぎ）
- Plan Mode（計画フェーズと実装フェーズのセッション分離）
- 手動でのコンテキストリセット

**検索インデックス系（情報取得の改善）**
- Serena（LSPを活用したコードベースのセマンティック検索）
- RAGベースのコンテキスト補完

これらのアプローチはいずれも「情報をどう圧縮するか」「必要な情報をどう取ってくるか」に焦点を当てており、**エージェントが「何が有効だったか」を選択的に記憶する機構を持たない**という共通の限界がある。

現状のLLMエージェントは成功・失敗をフラットにトークンとして積み上げるだけであり、過去の経験を再利用可能な形で保持・参照する仕組みを欠いている。特に「何をやってはいけないか」という失敗知識は、長期的なエージェント活動の質に直結するにも関わらず、既存手法では明示的に扱われていない。

---

## 2. リサーチクエスチョン

**RQ1（主）**  
LLMエージェントに連想記憶層（成功・失敗体験）を付与することで、長時間セッションにおけるコンテキスト劣化（Context Rot）を抑制できるか？

**RQ2（副）**  
成功体験のみ・失敗体験のみ・両方、の3条件で効果に差はあるか？どの組み合わせが最も有効か？

**RQ3（副）**  
フィードバックシグナル（インタラプト・rewind・修正指示）の種類と強度は、記憶エントリの品質と相関するか？

**RQ4（副）**  
ACMによる外部記憶の活用は、コンテキストウィンドウサイズへの依存度を低減するか？（コンテキストを意図的に制限した条件でも性能を維持できるか）

**RQ5（副）**  
提案手法はSerenaのような既存の検索インデックス手法と直交するか、あるいは競合するか？組み合わせた場合の効果は？

## 3. 提案手法：Associative Context Memory (ACM)

### 3.1 アーキテクチャ概要

```
ユーザーの指示
    ↓
[ACM Layer]  ←─── 連想メモリDB（永続化）
    ↓                ↑ 昇格/降格
Claude Code / Codex（実行エンジン）
    ↓
コンテキストウィンドウ（揮発）
```

ACMはMCPサーバーとして実装し、Claude CodeおよびCodexのエコシステムに乗せる。

### 3.2 連想メモリの構造

各エントリは**成功体験**と**失敗体験**の2種類を持つ：

**成功体験エントリ（"do this"）**
```json
{
  "type": "success",
  "trigger": "認証バグの修正、auth.py関連のエラー",
  "action": "FileSystemWatcherでの変更検知 → Diff表示の順で対応",
  "outcome": "テスト通過、手戻りなし",
  "retrieval_keys": ["auth", "FileSystemWatcher", "diff", "認証"],
  "signal_strength": 0.9
}
```

**失敗体験エントリ（"don't do this"）**
```json
{
  "type": "failure",
  "trigger": "認証バグの修正、auth.py関連のエラー",
  "action": "全ファイルを一括書き換えしようとした",
  "outcome": "インタラプト発生。直後の会話：「そうじゃなくて差分だけ変えて」",
  "retrieval_keys": ["auth", "一括書き換え", "差分"],
  "signal_strength": 0.95,
  "interrupt_context": "インタラプト後N=3ターンの会話をキャプチャ"
}
```

- **trigger**: どんな状況・問題だったか
- **action**: どういうアプローチをとったか
- **outcome**: 結果と、失敗の場合はインタラプト後の会話
- **retrieval_keys**: 連想検索に使うキーワード群
- **signal_strength**: フィードバックシグナルの強度（後述）
- **interrupt_context**: インタラプト後N턴の会話（失敗体験のみ）

### 3.3 フィードバックシグナルの強度序列

ユーザーからの暗黙的フィードバックを以下の強度で分類する：

```
強い ─── インタラプト（Ctrl+C / Esc連打）+ 直後の会話
             → 最も明確な「やめてほしかった」シグナル
             → 直後N턴に「なぜやめたか」が出てくる
         ↓
         rewind（/rewind または Esc×2）
             → 「なかったことにしたい」シグナル
         ↓
         修正指示（「違う」「やり直して」等のテキスト）
             → 言語化された不満
         ↓
         手戻りなし完了
             → 暗黙の成功シグナル
──── 弱い
```

この序列自体が論文の貢献の一つとなる：**LLMエージェントへの暗黙的フィードバックの分類と強度定義**。

インタラプト後の会話キャプチャはPhase 0で取得可能性を検証する。

### 3.4 検索・参照メカニズム

新しいタスク開始時に：
1. タスクの概要から retrieval_keys でベクトル検索
2. 類似する過去の成功体験を上位N件取得
3. 「この状況ではこのアプローチが有効だった」としてコンテキストの冒頭に挿入
4. 詳細情報は外部ファイル（ドキュメント）への参照として保持し、コンテキストを圧迫しない

## 4. 評価設計

### 4.1 実験条件

| 条件 | 内容 |
|-----|-----|
| Control | 素のClaude Code（コンテキスト管理なし） |
| Baseline-compact | Claude Code + auto-compact |
| Baseline-Serena | Claude Code + Serena |
| ACM-S | Claude Code + ACM（成功体験のみ） |
| ACM-F | Claude Code + ACM（失敗体験のみ） |
| ACM-SF | Claude Code + ACM（成功＋失敗両方） |
| ACM-SF + Serena | 組み合わせ |

ACM-S / ACM-F / ACM-SF の比較がアブレーションスタディとなり、RQ2に答える。

### 4.2 コンテキスト制限条件（RQ4用）

上記の各条件に対し、コンテキストウィンドウを人為的に制限した条件を追加：

| 制限 | 説明 |
|-----|-----|
| Full（200k） | 制限なし |
| Half（100k） | 通常の50% |
| Smart zone only（80k） | 〜40%に相当 |

「ACMがあればコンテキストを小さく保っても性能が維持されるか」を測定する。これが示せると、**モデルのスケールアップではなく記憶アーキテクチャの改善でContext Rotを解決できる**という強いクレームになる。

### 4.3 評価タスク

長時間セッションでのコンテキスト劣化が顕在化しやすいタスクを設計する：

**タスクA**: 複数ファイルにまたがるバグ修正（5〜10ファイル変更）  
**タスクB**: 既存コードベースへの機能追加（仕様書から実装まで）  
**タスクC**: リファクタリング（設計原則の一貫した適用）

各タスクを繰り返し実行し、2回目以降に連想メモリが参照可能な状態にする。

### 4.4 評価指標

| 指標 | 測定方法 | 対応RQ |
|-----|---------|-------|
| タスク完了率 | テスト通過率 or 人手評価 | RQ1 |
| 手戻り回数 | rewind・修正指示・インタラプトのカウント | RQ1, RQ3 |
| コンテキスト使用効率 | 完了時トークン使用量 / タスク複雑度 | RQ4 |
| セッション間の改善率 | 同一タスクの2回目以降のスコア変化 | RQ1, RQ2 |
| シグナル-成功相関 | 昇格シグナル強度 × 最終タスク成功率 | RQ3 |

**RQ1**: Control vs ACM-SF のセッション間改善率  
**RQ2**: ACM-S vs ACM-F vs ACM-SF の比較  
**RQ3**: インタラプト/rewind/修正指示のシグナル強度と記憶品質の相関  
**RQ4**: コンテキスト制限条件でのControl vs ACM-SF  
**RQ5**: Baseline-Serena vs ACM-SF + Serena

## 7. 関連研究との位置づけ

```
Context Rot (Chroma 2025)
  └─ 問題の記述・測定
      ↓
auto-compact / Plan Mode
  └─ 対症療法（圧縮・分割）
      ↓
Serena
  └─ 情報取得の改善（検索インデックス）
      ↓
Associative Context Memory（本研究）
  └─ 成功・失敗体験の選択的記憶・再利用
  └─ 暗黙的フィードバックシグナルの分類と強度定義
  └─ コンテキストウィンドウ依存からの脱却
     （上記すべてと直交するレイヤー）
```

本研究の貢献は3層ある：
1. **概念的貢献**: LLMエージェントへの暗黙的フィードバックシグナルの分類・強度序列の定義
2. **アーキテクチャ的貢献**: 成功・失敗体験を外部記憶に永続化するACMの設計
3. **実証的貢献**: Context Rot抑制・コンテキスト依存低減の効果測定

## 8. 未解決の問い（今後の検討事項）

- インタラプト後のキャプチャターン数N の最適値（多すぎるとノイズ、少なすぎると情報不足）
- 連想メモリの「忘却」をどう設計するか（古い体験が逆効果になるケース）
- プロジェクト固有の記憶 vs グローバルな記憶の分離
- マルチエージェント環境での共有メモリの可能性
- プライバシー・セキュリティ（体験エントリにコードが含まれる場合）
- 失敗体験が蓄積しすぎた場合の過学習的な問題
